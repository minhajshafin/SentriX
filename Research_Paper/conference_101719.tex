\begin{abstract}
This paper presents SentriX, a middleware-independent multi-stage security proxy for heterogeneous IoT protocols. SentriX unifies MQTT and CoAP behavioral detection through a fixed 33-dimensional feature abstraction (15 normalized behavioral dimensions + protocol identity + auxiliary features) without broker modification or protocol translation. A cross-protocol threat model covers volumetric, semantic, amplification, and slow-rate attacks across both protocol stacks. The implemented C++ proxy pipeline integrates rule-based filtering, ONNX-accelerated ML inference, and protocol-specific mitigation in a single in-line edge path. Using a custom dual-protocol testbed, we collect and label 11,209 behavioral observations across 6 attack classes and benign traffic. Four ML models are trained and evaluated using 5-fold stratified group cross-validation, with LightGBM achieving F1-macro of 0.5977 (accuracy 0.7796, 95\% CI [0.589, 0.607]). Live proxy validation over 101 runtime vectors confirms zero false positives at threshold 0.75, and a direct-vs-proxy latency check shows a mean overhead of 0.306 ms. Cross-protocol zero-shot transfer yields F1-macro $\approx$ 0.08, highlighting the generalization gap confirmed by Mann-Whitney U test between MQTT and CoAP anomaly score distributions (p = 0.0001). This work provides a measured baseline architecture, a reproducible dataset, and a complete C++ implementation for edge-native IoT anomaly detection across heterogeneous protocols.
\end{abstract}

\section{Introduction}
Modern IoT systems increasingly combine transport-diverse and semantics-diverse application protocols, especially MQTT and CoAP. Existing protection approaches are often constrained by one of three limitations: protocol specificity, broker coupling, or cloud-centralized detection paths that increase response latency. These limitations reduce applicability in edge deployments where in-line decisions, protocol correctness, and low overhead are mandatory.

This work addresses that gap through SentriX, a dual-protocol reverse proxy architecture that preserves protocol-native forwarding while unifying security reasoning across protocols. MQTT traffic remains MQTT over TCP, and CoAP traffic remains CoAP over UDP; no translation layer is introduced. Instead, SentriX maps protocol-specific observations to a normalized behavioral feature space that supports one shared machine learning model.

The main contributions are as follows:
\begin{itemize}
\item A formal protocol-normalized feature abstraction with 33 fixed dimensions (15 behavioral + 2 protocol ID + 16 auxiliary), frozen as a specification enabling reproducible cross-protocol evaluation.
\item A dual-protocol testbed and reproducible dataset generation pipeline with 11,209 labeled observations across 6 attack classes and 2 protocols.
\item A cross-protocol threat model grounding attacks in protocol-specific semantics for both MQTT and CoAP.
\item A fully implemented C++ reverse proxy with ONNX-runtime inference, validated end-to-end with 101 live vectors, zero false positives at the deployed threshold, and a measured mean proxy overhead of 0.306 ms.
\item ML evaluation showing F1-macro 0.5977 (LightGBM, 5-fold grouped CV) with statistical significance confirmed against MLP and LogReg; weak zero-shot cross-protocol generalization (mean gap 0.546) identified as a key open challenge rather than a solved deployment result.
\item An extensible architecture with a documented \texttt{IProtocolModule} C++ interface for adding new IoT protocols.
\end{itemize}

\section{Related Work and Motivation}
Prior IoT security systems can be grouped into: (i) protocol-specific detectors, (ii) broker-integrated plugins, (iii) network-level IDS, (iv) cloud-side anomaly detectors, and (v) protocol translators or gateways. Protocol-specific systems (e.g., MQTT IDS variants~\cite{mqtt_ml_ids}, MQTTGuard~\cite{mqttguard}, CoAP-focused defenses~\cite{coap_ids}) typically capture rich semantics but do not generalize across heterogeneous protocols. Broker plugins~\cite{broker_plugin} require broker-level integration and lack portability across broker implementations. Network IDS solutions such as Snort~\cite{snort} and Suricata~\cite{suricata} rely on packet or flow features that underrepresent application semantics, and do not support in-line per-protocol behavioral normalization. Cloud-side anomaly detection~\cite{cloud_iot_ml} introduces upstream connectivity dependencies and increased control-loop latency, making it unsuitable for edge-native mitigation. Protocol translators and IoT gateways~\cite{iot_gateway} violate strict protocol-preservation constraints in security-critical paths.

Table~\ref{tab:related} summarizes the key differentiators with representative prior work. The core gap is not simply multi-protocol support, but secure, in-line, protocol-preserving detection with shared behavioral learning across heterogeneous protocol semantics. We position SentriX as a baseline architecture that (1) unifies MQTT and CoAP behavioral features in a shared normalized representation, (2) deploys that representation in a standalone C++ reverse proxy with zero broker modification, and (3) quantifies cross-protocol generalization under controlled zero-shot transfer conditions.

\section{Threat Model}
\subsection{Adversary Capabilities}
The adversary can send arbitrary packets to the externally exposed ports of either proxy instance. The adversary can craft both valid and malformed protocol-native messages, including control-plane misuse and semantic abuse. No physical access to proxy hosts or backend brokers is assumed.

\subsection{Attack Taxonomy}
SentriX considers both cross-protocol and protocol-specific classes:
\begin{itemize}
\item \textbf{Volumetric abuse}: MQTT publish/connect floods; CoAP request floods.
\item \textbf{Protocol abuse}: malformed control fields, oversized payloads, invalid options, token misuse.
\item \textbf{Semantic abuse}: topic/resource squatting and payload injection.
\item \textbf{Slow-rate abuse}: MQTT SlowITe-like behavior; delayed CoAP observe or acknowledgment patterns.
\item \textbf{Amplification abuse}: MQTT QoS handshake exploitation; CoAP observe-based amplification.
\end{itemize}

\subsection{Trust Boundaries}
IoT clients are untrusted. Proxy-to-broker links are trusted (localhost or secured internal channel). MQTT and CoAP proxy paths are operationally independent except for shared detection model and metrics. The monitoring dashboard is read-only and authenticated.

\section{Protocol-Normalized Feature Framework}
\subsection{Design Principle}
SentriX separates protocol-native parsing from protocol-agnostic behavioral modeling. Let $\mathbf{x}^{(p)}$ denote a raw feature vector extracted from protocol $p \in \{\text{MQTT},\text{CoAP}\}$. A normalization function maps protocol-specific vectors into a shared representation:
\begin{equation}
\mathbf{z} = \Phi_p\left(\mathbf{x}^{(p)}\right), \quad \mathbf{z} \in \mathbb{R}^{15}.
\end{equation}

This shared vector is concatenated with protocol identity and protocol-specific auxiliary features to form the model input:
\begin{equation}
\mathbf{f} = [\mathbf{z} \;\|\; \mathbf{i}_p \;\|\; \mathbf{a}_{\text{MQTT}} \;\|\; \mathbf{a}_{\text{CoAP}}], \quad \mathbf{f} \in \mathbb{R}^{33}.
\end{equation}

\subsection{Frozen Normalized Dimensions}
The 15 normalized dimensions include message rate, inter-arrival time, payload size and entropy, resource path depth and entropy, reliability level abstraction, session duration, unique resource count, error rate, handshake complexity, subscription breadth, reconnection rate, payload-to-resource ratio, and protocol compliance score.

Each dimension has an explicit window policy and normalization rule (e.g., min-max scaling, clipped z-score, bounded ratio), frozen as Week 1 specification v1.0. This freeze ensures reproducible extraction for subsequent dataset generation and ablation studies.

\subsection{Extensibility}
The normalization interface is protocol-agnostic: adding a new protocol requires only a parser, raw feature extractor, and mapping $\Phi_p$ to the existing abstract dimensions (plus optional protocol-specific auxiliary features). The shared model pipeline remains unchanged.

\section{System Architecture}
\subsection{High-Level Design}
SentriX deploys two dedicated reverse proxies at the edge:
\begin{itemize}
\item MQTT proxy: TCP in-line path between MQTT clients and Mosquitto.
\item CoAP proxy: UDP in-line path between CoAP clients and Californium.
\end{itemize}
Both proxies feed a shared normalization and inference subsystem and export unified metrics to a monitoring dashboard.

\subsection{Shared Core Components}
The shared core includes:
\begin{itemize}
\item normalization engine,
\item cross-protocol rule evaluator,
\item ML inference runtime,
\item mitigation decision engine,
\item metrics collector and export interface.
\end{itemize}

\subsection{Protocol Modules}
Each protocol module implements: packet parsing, raw feature extraction, normalization mapping, protocol-specific rule checks, and protocol-specific mitigation dispatch. This modularity keeps protocol logic isolated while preserving a common detection path.

\section{Multi-Stage Detection Pipeline}
\subsection{Stage 1: Fast Rule Filtering}
Stage 1 performs low-overhead checks on normalized behavior (e.g., extreme message rate, anomalous payload entropy, reconnection bursts) plus protocol-specific checks (e.g., MQTT wildcard misuse, CoAP observe abuse indicators). This stage removes clearly malicious events early and limits inference load.

\subsection{Stage 2: Lightweight ML Inference}
Stage 2 uses a compact model family (e.g., tree ensembles or small MLP) over the 33-dimensional input vector. Model selection is based on per-protocol detection quality, macro F1, inference latency, and memory footprint. The joint model objective explicitly measures whether normalized behavioral dimensions support robust learning across both MQTT and CoAP.

\subsection{Stage 3: Adaptive Protocol-Aware Mitigation}
Mitigation decisions are produced from Stage 1 and Stage 2 outputs. Cross-protocol actions include forward, drop, and rate-limit. Protocol-specific actions include MQTT disconnect/subscription rejection and CoAP reset/observe rejection/response throttling. All decisions are logged with protocol and attack context for later analysis.

\section{Experimental Setup}
\subsection{Testbed Topology}
The evaluation uses a dual-path edge testbed with two independent protocol-native data planes:
\begin{itemize}
\item MQTT path: Paho clients $\rightarrow$ SentriX MQTT proxy $\rightarrow$ Mosquitto broker.
\item CoAP path: libcoap/aiocoap clients $\rightarrow$ SentriX CoAP proxy $\rightarrow$ Californium server.
\end{itemize}
Both proxy instances run on the same edge node and share the normalization, inference, and metrics subsystems, while preserving protocol-native forwarding and protocol-specific mitigation control.

\subsection{Traffic Generation}
For each protocol, benign and attack traffic are generated using scripted workloads.

\textbf{Benign MQTT workload:} periodic and bursty publish streams from simulated sensors (e.g., temperature, humidity, motion), varied topic depth, mixed QoS levels, and heterogeneous payload sizes.

\textbf{Benign CoAP workload:} GET/PUT/POST traffic over realistic resource trees, mixed CON/NON messaging, periodic Observe registrations, and block-wise transfers for larger payload scenarios.

\textbf{Attack MQTT workload:} publish flood, connection exhaustion, malformed control/payload injection, wildcard subscription abuse, SlowITe-like slow connect behavior, and QoS-2 handshake abuse.

\textbf{Attack CoAP workload:} high-rate request flood, Observe amplification, token exhaustion, malformed option injection, resource discovery abuse, confirmable flooding behavior, and block-wise transfer abuse.

\subsection{Data Collection and Labeling}
Each scenario is executed in labeled windows with synchronized workload metadata and proxy telemetry. Labels include protocol identifier, attack class, and attack-family tag. The pipeline records:
\begin{itemize}
\item raw protocol events and parser outcomes,
\item extracted protocol-specific raw features,
\item normalized 15-dimensional behavioral vectors,
\item protocol auxiliary feature vectors,
\item mitigation decisions and timestamps.
\end{itemize}
Ground-truth labels are produced by orchestrator scripts that control workload start/stop boundaries and scenario IDs. Label quality is validated by cross-checking orchestrator logs with proxy event timestamps.

\subsection{Feature Construction Workflow}
The end-to-end feature workflow is:
\begin{enumerate}
\item parse protocol-native packets/messages,
\item extract protocol-specific raw features,
\item apply protocol mapping $\Phi_p$ to produce normalized features,
\item append protocol identity and auxiliary dimensions,
\item export fixed 33-dimensional vectors with labels.
\end{enumerate}
Windowed features are computed with source-scoped sliding windows; per-message and aggregated features are emitted with aligned timestamps to support both online inference and offline training.

\subsection{Model Training and Selection}
Four candidate classifiers are trained on the combined MQTT+CoAP labeled dataset: LogisticRegression (baseline), RandomForest, MLP, and LightGBM. Training uses all 11,209 samples without validation hold-out. Evaluation uses 5-fold stratified group cross-validation, stratified by both attack class and protocol, with grouping by run-ID to prevent data leakage. Model selection prioritizes macro F1-score (primary metric), per-protocol detection quality, and inference latency. LightGBM is selected with F1-macro = 0.5977, serialized to ONNX format for C++ runtime inference (\texttt{lightgbm\_full.onnx}).

\subsection{C++ Proxy Integration}
The selected LightGBM ONNX model is loaded at proxy startup via ONNX Runtime C++ API. Each protocol module extracts the 33-dimensional feature vector and submits it to the shared inference pipeline. Inference is synchronous and in-line with the forwarding path. The MQTT proxy listens on port 1884 (forwarding to Mosquitto on 1883); the CoAP proxy listens on port 5684 (forwarding to Californium on 5683). Protocol-specific mitigation actions (rate-limit, drop, disconnect, observe-reject) are dispatched based on ML output combined with rule-stage decisions.

\subsection{Reproducibility and Artifacts}
The study produces: (1) labeled dataset CSV with 11,209 rows; (2) Python training and evaluation scripts; (3) ONNX model (\texttt{lightgbm\_full.onnx}); (4) feature specification document; (5) Dockerfile and docker-compose configuration; (6) simulation scripts for MQTT and CoAP benign and attack traffic; (7) metrics API server and Next.js monitoring dashboard. All artifacts are organized for independent verification.

Training on one protocol and testing on the other reveals substantially weaker performance. CoAP-only training tested on MQTT yields F1-macro = 0.0899 (vs. 0.5977 for grouped CV). \subsection{Statistical Significance}
Pairwise model comparisons on grouped CV (full feature set, n = 11,209) use a normal approximation to the difference in F1-macro scores. LightGBM vs. RandomForest yields $\Delta$F1 = +0.0008 (p = 0.91), indicating the two models are statistically equivalent and either is a valid deployment choice. LightGBM vs. MLP yields $\Delta$F1 = +0.0521 (p $<$ 0.001), and LightGBM vs. LogReg yields $\Delta$F1 = +0.2100 (p $<$ 0.001), both significant.

\subsection{End-to-End Live Proxy Inference}
The integrated C++ proxy was exercised over 101 runtime feature vectors (87 MQTT, 14 CoAP) collected from live proxy traffic. Anomaly scores range from 0.159 to 0.678 with all-protocol mean 0.218 and P95 0.458. At the deployed threshold of 0.75, zero vectors are flagged — confirming zero false positives in benign traffic. MQTT and CoAP anomaly score distributions differ significantly (Mann-Whitney U, p = 0.0001, Cohen's $d$ = 0.206), consistent with residual protocol-specific signal retained even after normalization. Bootstrap 95\% confidence intervals: MQTT mean 0.220 [0.197, 0.247]; CoAP mean 0.202 [0.188, 0.226]. Figure~\ref{fig:runtime_anomaly_distribution} shows the observed runtime score distributions and threshold margin.

\subsection{Interpretation}
The success of within-protocol learning (F1 $\approx$ 0.60) combined with failure of zero-shot transfer (F1 $\approx$ 0.08) indicates that protocol-specific auxiliary features and learned conditional boundaries dominate the decision function. The mean generalization gap of 0.546 quantifies the cost of cross-protocol deployment without domain adaptation. The statistically significant difference in runtime anomaly distributions (p = 0.0001) corroborates this: despite the normalized shared layer, MQTT and CoAP exhibit distinct behavioral signatures that the current model does not bridge. Feature drift analysis (legacy vs. behavioral vectors) identifies f06, f05, f04, f08, and f11 as the highest-drift dimensions (KL divergence $>$ 27.6), corresponding to rate, entropy, and reliability-level features most affected by the TCP-vs-UDP transport difference.

\section{Limitations and Future Work}
SentriX delivers a complete architecture and integrated system but has the following remaining limitations:
\begin{itemize}
\item \textbf{Cross-Protocol Generalization:} The mean generalization gap of 0.546 between in-protocol and cross-protocol F1 is the primary technical limitation. Future work should investigate adversarial domain adaptation, protocol-conditioned fine-tuning, and shared subspace alignment techniques to close this gap.
\item \textbf{Latency and Throughput Benchmarking:} The paper now reports direct-vs-proxy latency, but sustained throughput under heavy load and degradation curves are still missing. A dedicated profiling run with timestamped proxy logs is required before deployment-readiness claims can be made.
\item \textbf{Scale and Stress Testing:} The live evaluation used 101 vectors under benign conditions. Sustained concurrent attack traffic, mixed MQTT+CoAP stress at production rates, and degradation analysis under overload remain future work.
\item \textbf{Comparison with Baseline Systems:} No head-to-head comparison with deployed MQTT IDS, CoAP IDS, or network-level IDS systems (e.g., Suricata with IoT protocol parsers) was conducted in this work; the present study instead establishes internal baselines against direct forwarding and within-protocol models.
\item \textbf{Dataset Scale:} The labeled dataset of 11,209 observations covers 6 attack classes. Expanding to include slow-rate and amplification attacks under more realistic traffic conditions, and increasing CoAP coverage (currently 40\% of the dataset), would improve generalization.
\end{itemize}

\section{Conclusion}
SentriX presents a complete, middleware-independent multi-stage security proxy for heterogeneous IoT protocols. Through a protocol-normalized 33-dimensional behavioral feature abstraction, a single LightGBM model achieves F1-macro 0.5977 (accuracy 0.7796, 95\% CI [0.589, 0.607]) on 5-fold grouped cross-validation over 11,209 labeled observations spanning MQTT and CoAP attack classes. The C++ proxy integrates ONNX-accelerated inference end-to-end, confirmed with zero false positives over 101 live runtime vectors at threshold 0.75 and a measured mean latency overhead of 0.306 ms.

The principal research finding is a large cross-protocol generalization gap (mean $\Delta$F1 = 0.546): protocol-normalized behavioral features support strong within-protocol detection but do not transfer zero-shot across MQTT and CoAP. Statistical analysis confirms that MQTT and CoAP anomaly score distributions are significantly different (Mann-Whitney p = 0.0001), and feature drift analysis identifies rate, entropy, and reliability-level dimensions (f04--f08, f11) as the primary sources of cross-protocol divergence. These findings establish the paper as a benchmark study of the normalization framework and the transfer gap, not a claim that zero-shot detection is solved.

SentriX is released as an end-to-end reproducible artifact including C++ source, Python training pipeline, ONNX model, labeled dataset, traffic simulators, Docker infrastructure, and a real-time monitoring dashboard.
